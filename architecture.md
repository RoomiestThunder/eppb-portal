# ЕППБ — архитектурный подход (MVP)

Краткое описание архитектуры, требуемое по условиям конкурса. Документ описывает
принципы, а не полную техническую спецификацию — этого достаточно, чтобы показать,
как решение масштабируется на 70+ услуг без переписывания кода под каждую из них.

## 1. Идея: услуга — это данные, а не код

Ключевое архитектурное решение MVP: **сервис (услуга), её этапы, шаги, поля,
условия ветвления и формулы расчёта хранятся как данные в базе, а не как код**.

```
Organization ──< Service ──< ServiceStage ──< ServiceStep ──< FormField
                                                                  │
                                                     visibilityRule (JSON)
                                                     formula (выражение)
                                                     lookupCode (справочник)
```

Два независимых движка читают одну и ту же схему:

- **Клиентский визард** (`ApplicationWizard`) — рендерит шаги предпринимателю,
  подставляет значения, вычисляет формулы, скрывает/показывает поля.
- **Административный конструктор** (`ServiceBuilder`) — тот же рендер, но в
  режиме редактирования схемы.

Общая логика вынесена в `src/lib/ruleEngine.ts` — оценка условий показа поля
(`visibilityRule`, дерево `and/or/not` над операторами сравнения) и вычисление
`CALCULATED`-полей через собственный безопасный арифметический парсер (без
`eval`/`Function`, т.к. формулы — это пользовательские данные, а не доверенный код).

Добавление 71-й меры поддержки не требует деплоя: администратор или автор услуги
создаёт запись `Service` через конструктор, наполняет её этапами/полями/условиями
и публикует. Код приложения не меняется.

## 2. Слои системы

```
┌─────────────────────────────────────────────────────────────┐
│  Клиентский контур        │  Административный контур         │
│  главная / каталог /      │  конструктор услуг / заявки /     │
│  карточка услуги / визард │  справочники / аналитика / карта  │
│  / личный кабинет / AI    │  / инструменты / интеграции       │
├─────────────────────────────────────────────────────────────┤
│              Rule engine (visibility + formulas)              │
├─────────────────────────────────────────────────────────────┤
│   Прикладной слой: Next.js App Router (страницы + API-роуты)  │
├─────────────────────────────────────────────────────────────┤
│         Слой данных: Prisma ORM → БД (SQLite в MVP)           │
├─────────────────────────────────────────────────────────────┤
│   Интеграционный слой (mock): eGov IDP · ГБД ИИН/БИН · ЭЦП ·  │
│   BPM дочерних организаций · обмен документами · уведомления  │
└─────────────────────────────────────────────────────────────┘
```

Интеграционный слой — единая точка, через которую портал общается с внешним
миром (`src/lib/integrations.ts`). Каждый вызов логируется в `IntegrationLog`
и виден в админ-разделе «Интеграции». В продуктивном контуре этот слой
подключается к **Единой интеграционной шине Холдинга**: меняется реализация
внутри модуля, вызовы из остального кода не меняются.

## 3. Почему это масштабируется на 70+ услуг

- **Новая услуга = запись в БД**, не новый код и не новый деплой.
- **Новая дочерняя организация** = запись `Organization` (цвет бренда, код) —
  используется каталогом услуг, аналитикой и картой проектов одинаково.
- **Новый справочник** (`Lookup`) добавляется через админку и сразу доступен
  любому полю типа «Справочник» в любой услуге.
- **Новая роль** — модель `UserRole` уже разделяет предпринимателя,
  администратора и автора услуги; добавление, например, роли «модератор ДО»
  — это новое значение enum + условие доступа, без переписывания страниц.
- **Условная логика и расчёты** не жёстко закодированы под конкретную услугу —
  это интерпретируемые данные (JSON-условие, строка-формула), поэтому сложность
  конкретной услуги (как «Вагоны в лизинг», 2 этапа/5 шагов/условные документы)
  не тянет за собой рост кодовой базы.
- **Интеграции подключаются точечно**: у каждой услуги/оргии в перспективе может
  быть свой набор коннекторов (BPM конкретной ДО, свой справочник и т.д.) —
  расширение интеграционного слоя не затрагивает конструктор и рендер форм.

## 4. Что заменяется при переходе от MVP к продуктиву

| Компонент MVP | Продуктивный аналог |
|---|---|
| SQLite (файл) | Управляемая СУБД Холдинга (PostgreSQL и т.п.), при необходимости — с разделением по ДО |
| Mock eGov IDP (демо-вход по ролям) | Реальный eGov IDP (OIDC/SAML) |
| Mock ЭЦП | Интеграция с сервisами НУЦ РК |
| Mock BPM submit/status | Единая интеграционная шина → BPM конкретной дочерней организации |
| Rule-based AI-движок | Тот же интерфейс, но с реальным вызовом LLM/ML-сервиса под капотом — вызывающий код не меняется |
| Каталог аналитики со ссылкой/заглушкой embedding | Реальный iframe/SDK встраивания дашбордов после согласования CSP/X-Frame-Options с ДО |
| Единый мутационный API конструктора | При росте нагрузки — разбивается на отдельные REST/RPC-ресурсы с очередями для тяжёлых операций публикации |

## 5. AI-компонент

Реализован как rule-based движок (`src/lib/ai.ts`) с точки входа:
подбор услуги по ключевым словам, объяснение услуги простым языком, проверка
полноты заявки перед отправкой (клиентский путь) и генерация черновика
структуры услуги по свободному описанию (помощь автору услуги). Решение
намеренно не зависит от внешнего API — демо-стенд работает без ключей и
сетевых вызовов. Интерфейс модуля спроектирован так, что рантайм-логику можно
заменить на вызов внешней языковой модели, не меняя вызывающий код на
страницах и в API-роутах.

## 6. Контрольный кейс как проверка конструктора

Обе контрольные услуги («Приобретение вагонов в лизинг» и «Субсидирование
животноводства») реализованы **только через конструктор** — как записи в БД
(см. `prisma/seed.ts`), без отдельного кода под каждую. Это включает: два этапа
(первичная заявка + расширенные данные), пошаговые формы, расчётные поля
(итоговая стоимость, аванс, ежемесячный платёж / сумма субсидии), условные
поля и документы (например, бизнес-план требуется только при партии от 10
вагонов; документ нерезидента — только для нерезидентов), справочники
(типы вагонов, виды животноводства, регионы) и предзаполнение данных по БИН.
